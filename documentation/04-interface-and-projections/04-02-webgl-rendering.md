# 4.2: WebGL Hypergraph Rendering Engine and Spatial Layout Mechanics

To visualize and interact with internet-scale hypergraphs without performance degradation, Holds implements a custom GPU-accelerated rendering engine built on WebGL2 / WebGPU. The rendering subsystem bypasses conventional DOM-based UI nodes entirely, using instanced buffer streaming to project nested grouping membranes, $n$-ary hyperedges, and multi-dimensional node identities onto a continuous 2D/3D spatial canvas in real time.

---

## 4.2.1: GPU-Accelerated Hypergraph Rendering Pipeline

The rendering pipeline is designed to maintain 60 FPS performance when displaying subgraphs containing hundreds of thousands of active elements. It achieves this by coupling the Stage 0 Memory Arena directly to GPU hardware buffers via WebAssembly (`WASM`) shared memory.

```text
 [ Arena Memory (WASM) ]
          |
          v (Zero-Copy Buffer View)
 [ WebGL2/GPU Buffers ] ===> [ Instanced Vertex Shaders ] ===> [ Spatial Fragment Canvas ]
          ^                                  ^
          |                                  |
 [ Uniform Spatial Index ]          [ Dynamic Membrane LUT ]

```

### 4.2.1.1 Zero-Copy WASM-to-GPU Buffer Streaming

1. **Shared Memory Mapping:** The node positions, adjacency indices, and boundary constraints calculated by the layout physics engine reside within a contiguous `SharedArrayBuffer` managed by WebAssembly.
2. **Direct VBO Uploads:** Instead of serializing graph data into JavaScript objects before rendering, the engine streams byte slices directly from WASM linear memory into GPU Vertex Buffer Objects (VBOs) via `gl.bufferSubData`.
3. **Instanced Node and Edge Drawing:** Nodes, hyperedges, and membrane contours are rendered using instanced array drawing (`gl.drawArraysInstanced`). A single draw call renders up to $10^5$ nodes using instanced quad transforms, drastically reducing CPU-to-GPU draw call overhead.

### 4.2.1.2 Multi-Layered Shader Pipeline

The shader pipeline separates rendering responsibilities across three dedicated draw passes:

* **Pass 1: Membrane Surface & Depth Shader:** Renders semi-transparent, tinted polygon meshes representing grouping boundaries. It uses signed distance fields (SDF) to draw crisp membrane perimeters and interior potential fields at any scale.
* **Pass 2: Hyperedge & Adjacency Mesh Shader:** Renders binary edges as anti-aliased vector lines and $n$-ary hyperedges as filled convex hulls or Bézier ribbons linking arbitrary $N$-node tuples.
* **Pass 3: Node & Identity Glyph Shader:** Renders node anchors, multi-dimensional hash halos ($H_{id}$ badges), and text glyphs using instanced Texture Atlases and MSDF (Multi-channel Signed Distance Fields).

## 4.2.2: Force-Directed Layout Mechanics over Nested Membranes

Standard force-directed graph algorithms (e.g., Fruchterman-Reingold or Barnes-Hut) assume flat 2D topologies. In Holds, layout mechanics must respect non-Euclidean boundary containment, hierarchical nesting, and variable membrane spin ($+1$ vs. $-1$).

The layout engine employs a modified **Hierarchical Barnes-Hut Physics Engine with Spatial Membrane Potentials**.

### 4.2.2.1 The Force Vector Equation

The total net force $\mathbf{F}_{\text{total}}(v)$ acting on an atom or node $v$ is given by:

$$\mathbf{F}_{\text{total}}(v) = \mathbf{F}_{\text{repulsion}}(v) + \mathbf{F}_{\text{attraction}}(v) + \mathbf{F}_{\text{membrane}}(v) + \mathbf{F}_{\text{spin}}(v)$$

Where:

* **Repulsive Force ($\mathbf{F}_{\text{repulsion}}$):** Standard inverse-square electrostatic repulsion evaluated across all nodes sharing the same immediate parent scope.
* **Attraction Force ($\mathbf{F}_{\text{attraction}}$):** Hooke's-law spring forces applied along incident $n$-ary hyperedges, pulling connected nodes into spatial proximity.
* **Membrane Potential Force ($\mathbf{F}_{\text{membrane}}$):** A containment force derived from a continuous potential field $U_{\text{membrane}}(v)$ generated by enclosing grouping boundaries.

### 4.2.2.2 Potential Fields for Boundary Containment

To keep child nodes strictly contained inside their parent membrane boundary $M$, $M$ acts as a soft-wall potential well:

$$U_{\text{membrane}}(v) = \begin{cases}  0, & \text{if } \text{dist}(v, \partial M) > r_{\text{threshold}} \\ k_{\text{wall}} \cdot \left( \frac{1}{\text{dist}(v, \partial M)} \right)^2, & \text{if } v \text{ approaches } \partial M \text{ from inside} \end{cases}$$

$$\mathbf{F}_{\text{membrane}}(v) = -\nabla U_{\text{membrane}}(v)$$

As node $v$ approaches the inner perimeter $\partial M$, the potential gradient generates a sharp repulsive vector pushing $v$ back toward $M$'s center of mass, preventing boundary breaches.

### 4.2.2.3 Non-Orientable Inversion Forces (Spin -1)

When a membrane's orientation bit is set to Spin `-1` (Klein Bottle topology), the potential field gradient inverts ($\mathbf{F}_{\text{membrane}} \to -\mathbf{F}_{\text{membrane}}$). The membrane perimeter becomes permeable, allowing the layout engine to smoothly draw interior nodes overflowing into the parent scope to visually reflect self-referential topology.

## 4.2.3: Real-Time Spatial Layout Stabilization and Delta Updates

When a structural rewrite rule $L \implies R$ occurs, applying a full force-directed layout re-calculation from scratch causes catastrophic visual jumpiness, destroying the user's mental map of the codebase.

Holds enforces **Incremental Layout Stabilization** to maintain spatial continuity across state mutations.

### 4.2.3.1 Anchor-Preserving Localized Relaxation

When a localized rewrite $L \implies R$ mutates a target region:

1. **Unmodified Node Freezing:** Nodes belonging to $L \cap R$ have their positions locked in spatial coordinates ($\text{velocity} = 0$, $\text{mass} = \infty$).
2. **New Node Initial Positioning:** Newly created nodes ($R - L$) are initialized at the center of mass of the matched LHS pattern, offset by randomized low-magnitude jitter vectors.
3. **Localized Physics Bounding Box:** Force calculations are restricted to a localized spatial bounding box enclosing the modified region $R$. The rest of the hypergraph remains completely stationary.
4. **Smooth Simulated Annealing:** A high-damping friction coefficient is applied locally, relaxing the new nodes into an energy-minimum layout within 300 milliseconds.

```text
  [ Unmodified Graph (Frozen) ] --------+
                                        |
  [ Mutated Region R ]                  | ---> Local Physics Relaxation
  +----------------------------------+  |      (Only new nodes update)
  |  New Node 1 ---> (Relaxing...)   |  |
  |  New Node 2 ---> (Relaxing...)   |  |
  +----------------------------------+  |
                                        |
  [ Unmodified Graph (Frozen) ] --------+

```

## 4.2.4: Instanced Rendering of N-Ary Hyperedges and Membrane Enclosures

Displaying relationships that connect three or more nodes simultaneously requires specialized visual geometry rather than simple 2D line segments.

### 4.2.4.1 Convex Hull and Ribbon Generation

For an $n$-ary hyperedge linking $N$ nodes $\{v_1, v_2, \dots, v_N\}$:

* **Small $N$ ($N = 3, 4$):** The GPU geometry shader computes a smooth 2D Bézier patch or convex polygon whose vertices anchor to the instantaneous positions of $v_1 \dots v_N$.
* **Large $N$ ($N \ge 5$):** The engine computes an enclosing convex hull using a fast GPU parallel Graham-Scan pass. The interior surface is rendered as a semi-transparent, color-coded mesh, while the perimeter is outlined with a glowing contour line.

### 4.2.4.2 Visualizing Opaque vs. Permeable Membranes

Grouping membranes are visually styled according to their semantic attributes:

* **Standard Membrane:** Rendered as a smooth, rounded rectangle or organic blob with an interior translucent fill.
* **Opaque Membrane (`meta::opaque`):** Rendered with a solid, light-blocking perimeter wall, signaling visually that internal structures are isolated from external pattern matching and vacuum checks.
* **Residue Ghost Membrane (`sys::residue`):** Rendered in low-opacity, desaturated grey with a dashed boundary, visually distinguishing historical/abandoned state from active runtime topology.
