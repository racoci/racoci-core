# 6.1: Stage 0 Memory Arena Allocator and Structural Indexing

At the lowest physical layer of the Holds runtime lies the **Stage 0 Memory Arena Allocator**. Operating substrate performance depends on eliminating OS-level dynamic heap allocations (`malloc`/`free`) and avoiding non-deterministic Garbage Collection (GC) pauses during high-frequency graph rewriting. Holds achieves zero-cost memory management by organizing all atoms, hyperedges, membranes, and structural metadata inside a flat, contiguous, immutable linear byte arena.

---

## 6.1.1: Linear Arena Layout and Monotonic Allocation Mechanics

The Stage 0 Arena is instantiated as a contiguous linear byte array mapped directly into WebAssembly (`WASM`) linear memory or native virtual memory pages.

```text
 [ Arena Base Pointer ]
 +---------------------+---------------------+---------------------+---------------------+
 |  Atom Header #1     |  Hyperedge Header   |  Membrane Header    |  Unallocated Space  |
 |  [ 32 Bytes ]       |  [ 64 Bytes ]       |  [ 128 Bytes ]      |  (Free Arena Range) |
 +---------------------+---------------------+---------------------+---------------------+
 0x00000000            0x00000020            0x00000060            0x000000E0 ----> Bump Pointer

```

### 6.1.1.1 Monotonic Bump Pointer Allocation

Allocation within the arena operates via atomic bump-pointer arithmetic:

1. When a new atom, hyperedge, or membrane boundary is allocated during a rewrite step $L \implies R$, the allocator requests $B$ bytes.
2. The bump pointer $P_{\text{bump}}$ is incremented atomically using a single Compare-And-Swap (CAS) CPU instruction:

$$P_{\text{new}} = P_{\text{bump}} + B$$


3. The allocation complexity is strictly $O(1)$ and lock-free across concurrent worker threads.

### 6.1.1.2 Immutable Memory Cells

Once written into the arena, memory cells are **strictly immutable**. Nodes and adjacencies are never updated in-place. State changes generate new byte blocks at the current bump pointer location, updating parent pointers via Merkle-tree root mutations. Immutability guarantees thread safety without mutex locks and enables instant, zero-copy rollback to previous state offsets.

---

## 6.1.2: Flat 32-Bit/64-Bit Index Pointers vs. Hardware Memory Pointers

Traditional graph databases store relationships as 64-bit raw virtual memory pointers (`0x7FFF...`). This model breaks cross-process synchronization, prevents zero-copy network serialization, and consumes excessive memory bandwidth.

Holds replaces raw pointers with **Flat Arena Index Pointers** ($I_{\text{arena}}$).

```text
 32-Bit Index Pointer Layout:
 [ 2 Bits: Entity Type ] [ 3 Bits: LoD Slice ] [ 27 Bits: Relative Byte Offset ]

```

### 6.1.2.1 Relative Offset Indexing

An index pointer $I_{\text{arena}}$ encodes the relative offset from the base address of the Stage 0 Arena rather than a physical hardware memory address:

$$\text{Address}_{\text{physical}} = \text{Base}_{\text{arena}} + \left( I_{\text{arena}} \ll \text{Shift} \right)$$

* **Cross-Environment Compatibility:** The entire memory arena can be serialized directly to disk or transmitted over WebSockets/QUIC networks to remote worker nodes. The receiving node maps the buffer into its local memory space at an arbitrary base address without requiring pointer re-basing or pointer-swizzling passes.
* **Compact Memory Footprint:** By enforcing byte alignment, a 32-bit index pointer can address up to 16 Gigabytes of contiguous hypergraph structures inside a single arena partition.

---

## 6.1.3: Continuous Merkle-DAG Deduplication and Lookup Tables

To prevent memory inflation caused by immutability, the Stage 0 Allocator incorporates an inline **Structural Deduplication Engine**.

### 6.1.3.1 Inline Hash Interception

When a thread prepares to allocate a new subgraph block $B$:

1. The allocator calculates $B$'s pure topological hash $h_{\text{topo}}(B)$ using BLAKE3 SIMD instructions.
2. It queries a lock-free, concurrent Robin Hood Hash Table indexing active arena addresses by $h_{\text{topo}}$.
3. **Hit Case:** If an identical structural block $B'$ already exists in the arena ($h_{\text{topo}}(B') = h_{\text{topo}}(B)$), the allocator discards allocation $B$ and returns the existing index pointer $I_{\text{arena}}(B')$, achieving $O(1)$ structural deduplication.
4. **Miss Case:** If no match exists, $B$ is written at $P_{\text{bump}}$, and its index is inserted into the hash table.

```text
  Thread Request Allocation B
               |
               v
  Compute h_topo(B) via BLAKE3
               |
               v
  Look up in Arena Merkle-DAG Table
        /             \
  (Match Found)    (No Match)
      /                 \
  Return Pointer      Write to P_bump &
  to Existing B'      Update Hash Table

```

## 6.1.4: Epoch-Based Generational Compaction without GC Pauses

Because memory cells are immutable, old or replaced subgraphs gradually become unreachable (orphaned) as parent pointers mutate. Rather than running a stop-the-world tracing garbage collector, Holds uses **Epoch-Based Generational Arena Compaction**.

### 6.1.4.1 Arena Epochs and Pinning

1. Memory is partitioned into generational arena segments ($E_0, E_1, E_2, \dots$).
2. Active execution threads register an epoch pin when reading or transforming a subgraph region.
3. When an entire arena segment $E_k$ consists predominantly of orphaned nodes (determined via low-overhead reference counters on outer membrane boundaries), $E_k$ is flagged for compaction.

### 6.1.4.2 Zero-Copy Sliding Compaction

1. A background worker thread allocates a new contiguous segment $E_{k+1}$.
2. It copies only live root membranes and active subgraphs reachable from global roots into $E_{k+1}$, compacting nodes into contiguous memory addresses.
3. Once all threads unpin epoch $E_k$, the entire memory range of $E_k$ is reclaimed via a single $O(1)$ system call (`munmap` / WASM memory shrinking) without walking individual dead objects.

# 6.2: Universal State Synchronization Bus and Lock-Free Concurrency

In a multi-core or distributed environment, multiple rewrite rules, UI render loops, and background indexing agents attempt to inspect and modify the shared hypergraph substrate simultaneously. Holds implements a **Universal State Synchronization Bus** operating over shared memory channels (`SharedArrayBuffer` in WebAssembly or atomic shared memory pages in native systems) to achieve lock-free concurrent execution.

---

## 6.2.1: Lock-Free Atomic Root Pointer Mutations

All state in Holds descends from a single global root index pointer $R_{\text{root}}$ anchoring the top-level membrane boundary. State transitions are governed by lock-free **Atomic Compare-And-Swap (CAS)** operations over $R_{\text{root}}$.

```text
  Worker Thread A                       Global Root Pointer R_root                    Worker Thread B
  ----------------                      --------------------------                    ----------------
  1. Read R_old = 0x01                               |                                1. Read R_old = 0x01
  2. Compute Rewrite L => R                          |                                2. Compute Rewrite L' => R'
  3. Produce R_new_A = 0x02                          |                                3. Produce R_new_B = 0x03
  4. CAS(R_root, 0x01, 0x02)                         |                                4. CAS(R_root, 0x01, 0x03)
            |                                        |                                          |
     [ SUCCESS ] -----------------------------> 0x02                                            |
                                                     | <--------------------------------- [ FAIL: R_root != 0x01 ]
                                                     |                                    (Retry with R_old = 0x02)

```

### 6.2.1.1 Optimistic Concurrency Control (OCC) Pipeline

1. **Read Phase:** Worker threads read a snapshot of target subgraphs anchored at $R_{\text{old}}$ without acquiring read locks.
2. **Evaluate Phase:** Threads execute rewrite rules $L \implies R$ locally, allocating newly created nodes at their local bump-pointer offsets in shared arena space.
3. **Commit Phase:** Threads attempt to update $R_{\text{root}}$ from $R_{\text{old}}$ to $R_{\text{new}}$ using atomic CAS:

$$\text{CAS}(R_{\text{root}}, R_{\text{old}}, R_{\text{new}})$$


4. **Conflict Resolution:** If another thread mutated $R_{\text{root}}$ during step 2, the CAS fails. The losing thread discards its root pointer attempt, re-reads $R_{\text{root}}$, and re-evaluates the rewrite over the updated topology.

---

## 6.2.2: SharedArrayBuffer Ring-Buffer Delta Queues

To notify UI projections, layout engines, and remote peers of state changes without copying graph structures, the Synchronization Bus uses a high-throughput **SMC (Single-Producer Multi-Consumer) Ring Buffer**.

```text
 Ring Buffer Memory Array (SharedArrayBuffer):
 +-------------------+-------------------+-------------------+-------------------+
 | Delta Event #101  | Delta Event #102  | Delta Event #103  | Empty Slot        |
 | [ Mutated Offset ]| [ Mutated Offset ]| [ Mutated Offset ]| (Head Pointer)    |
 +-------------------+-------------------+-------------------+-------------------+
 Tail (Reader 1)     Tail (Reader 2)                         Head (Writer)

```

### 6.2.2.1 Structure of Delta Events

When a commit succeeds, the winning thread writes a compact, fixed-size **Delta Event** into the ring buffer:

$$\Delta_{\text{event}} = \left\langle T_{\text{commit}}, I_{\text{target\_membrane}}, h_{\text{topo}}(L), h_{\text{topo}}(R), \text{Offset}_{\text{diff}} \right\rangle$$

* **Zero-Copy Notification:** The delta event contains no node or edge payloads. It transmits arena index offsets and structural hash deltas.
* **Subscriber Consumption:** Listening components (such as the WebGL Render Loop running at 60 FPS) read delta events directly from shared memory, fetch modified node coordinates from the specified arena offsets, and update GPU VBO buffers in $O(1)$ time.

## 6.2.3: CRDT-Driven Pushout Merging over Non-Interfering Domains

When parallel worker threads modify spatially distinct regions of the hypergraph substrate, forcing one thread to retry its CAS due to a root pointer conflict is inefficient. Holds integrates **Conflict-Free Replicated Data Type (CRDT)** semantics into Double Pushout (DPO) category rewriting.

### 6.2.3.1 Commutative Parallel Pushouts

Let $r_1: L_1 \implies R_1$ and $r_2: L_2 \implies R_2$ be two rewrite rules operating on target hypergraph $G$:

```text
                      Target Hypergraph G
                         /           \
                        /             \
                       v               v
                Result G_1           Result G_2
                       \               /
                        \             /
                         v           v
                    Merged Substrate State M

```

If the matched subgraphs $m_1(L_1)$ and $m_2(L_2)$ share no common nodes or hyperedges outside invariant interface boundaries ($m_1(L_1) \cap m_2(L_2) \subseteq K_1 \cap K_2$), the rewrites are **Spatially Parallel Independent**.

### 6.2.3.2 Automatic Pushout Synthesis

Instead of rejecting the second thread's commit, the Synchronization Bus computes the **Categorical Pushout Union** $M = G_1 \sqcup_G G_2$:

1. The bus combines structural diffs $\Delta_1 = R_1 - L_1$ and $\Delta_2 = R_2 - L_2$.
2. It executes a dual-pushout join directly in the memory arena, producing unified state $M$ without re-evaluating rule $r_2$ from scratch.
3. This achieves lock-free, non-blocking parallel execution across arbitrary multi-core processors.

# 6.3: Distributed Substrate Protocol and Internet-Scale Replication

To scale the Holds continuous structural environment beyond a single machine, the architecture extends its hypergraph mechanics across peer-to-peer networks. In this model, distributed network nodes do not exchange JSON messages or REST requests; they synchronize slices of a single global, internet-scale hypergraph substrate using boundary membrane partitioning, Merkle Mountain Ranges (MMR), and gossip protocols.

---

## 6.3.1: Boundary Membrane Partitioning and P2P Sharding

In traditional databases, sharding requires partitioning tables by explicit keys or hash ranges. In Holds, the hypergraph shards naturally along **Grouping Membranes**.

```text
 [ Global Network Substrate ]
 +-----------------------------------------------------------------------+
 |  Peer Node A Scope                                                    |
 |  +----------------------------------+                                 |
 |  | Membrane M_1                     |                                 |
 |  |  (a) ----> (b)                   |                                 |
 |  +-------|--------------------------+                                 |
 |          |                                                            |
 |          +======== [ Cross-Network Hyperedge ]                        |
 |          |                                                            |
 |  +-------|--------------------------+                                 |
 |  | Membrane M_2                     |                                 |
 |  |  (c) ----> (d)                   |                                 |
 |  +----------------------------------+                                 |
 |  Peer Node B Scope                                                    |
 +-----------------------------------------------------------------------+

```

### 6.3.1.1 Membrane Sharding Rule

1. Any grouping membrane $M$ tagged with a public network boundary attribute (`net::shared`) acts as an autonomous network shard.
2. A peer node $P_1$ can host membrane $M_1$ locally while holding remote ghost references to membrane $M_2$ hosted on peer node $P_2$.
3. Cross-network hyperedges link nodes across machine boundaries by referencing remote index hashes ($h_{\text{full}}$) rather than local memory offsets.

### 6.3.1.2 Opaque Boundaries as Network Security Isolators

Membranes marked as opaque (`meta::opaque`) prevent remote peers from inspecting internal subgraphs during pattern-matching queries. The remote peer can verify $M$'s canonical hash vector $H_{id}(M)$ and interface ports, but cannot traverse $M$'s interior nodes, providing native privacy and security isolation across distributed trust boundaries.

---

## 6.3.2: Merkle Mountain Range (MMR) Synchronization Protocol

When two distributed peers establish a WebRTC or QUIC transport connection, they synchronize state differentials efficiently without transferring entire graph models.

```text
  Peer A Peak Digest List                          Peer B Peak Digest List
  [ H_1, H_2, H_3 ]                                [ H_1, H_2, H_4 ]
          |                                                |
          +================ ( Compare Peaks ) =============+
                                   |
                      Discrepancy Found at H_3 vs H_4
                                   |
                     Traverse Sub-Membrane Branches
                                   |
                    Isolate & Stream Delta Subgraph

```

### 6.3.2.1 State Synchronization Algorithm

1. **Peak List Exchange:** Each peer maintains an append-only Merkle Mountain Range (MMR) logging all local topological commits. Peers exchange top-level peak hash lists.
2. **Binary Divergence Search:** Peers perform logarithmic binary search down the MMR tree to locate the exact transaction step where their topological states diverged.
3. **Differential Subgraph Extraction:** The source peer queries its Stage 0 Arena using the divergent root hash, extracts the differential subgraph $\Delta H = H_{\text{local}} - H_{\text{remote}}$, and serializes the raw arena byte block.
4. **Zero-Copy Ingestion:** The receiving peer writes the incoming byte block directly into its local Stage 0 Arena and executes an inline Merkle-DAG deduplication pass, integrating the remote state into its local substrate in $O(\vert{}\Delta H\vert{})$ time.

---

## 6.3.3: Cryptographic Provenance Vectors and Provenance Tracking

To prevent malicious or corrupted state injections in a decentralized peer-to-peer network, every state transition in Holds carries immutable **Cryptographic Provenance Vectors** anchored inside the `sys::provenance` metadata namespace.

### 6.3.3.1 The Provenance Chain

When a distributed node executes a rewrite $L \implies R$, the resulting right-hand side structure $R$ is signed with a cryptographic provenance vector:

$$\text{Prov}(R) = \text{Sign}_{\text{Ed25519}}\left( \text{PeerID}, h_{\text{full}}(L), h_{\text{full}}(R), \text{Timestamp}_{\text{vector}}, h_{\text{rule\_identity}} \right)$$

```text
  [ Generated Subgraph R ]
         |
         +---> [sys::provenance]
                     |
                     +-- Peer Identity (Ed25519 Public Key)
                     +-- Cause Hash: h_full(L)
                     +-- Effect Hash: h_full(R)
                     +-- Applied Rule: h_type(Rule)
                     +-- Signature Digest

```

### 6.3.3.2 Distributed Auditability

* **Tamper Evidence:** Because the cause hash $h_{\text{full}}(L)$ links directly to the prior state, any node can independently verify the execution validity of rule $L \implies R$ by re-running the pattern matcher against $L$.
* **Bilinear Proof Verification:** Invalid rewrites (such as an unauthorized peer attempting to modify a protected network membrane) fail signature and rule-validation checks across the gossip network. The invalid state update is rejected by honest peers, maintaining global substrate consistency without relying on centralized servers or consensus bottlenecks.
